<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| current benchmark -> saved baseline benchmark history | The runtime compares fresh benchmark evidence against persisted baseline-side evidence under `.quantum/benchmarks/history/`. |
| CLI threshold flags -> runtime evaluator | User-supplied policy thresholds and booleans drive accept/reject behavior. |
| policy-bearing benchmark report -> shell / JSONL | CI will rely on FluxQ’s verdict and exit behavior instead of wrapper logic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | S/T | baseline benchmark evidence loading | mitigate | Resolve the saved workspace baseline first, then require matching subject identity from persisted benchmark history before any metric thresholds run. |
| T-04-02-02 | T | incomparable or missing backend evidence | mitigate | Fail closed with explicit `benchmark_backend_missing:*` or `benchmark_backend_incomparable:*` reason codes before threshold math. |
| T-04-02-03 | T | benchmark policy input | mitigate | Reject negative thresholds as `invalid_benchmark_policy` and keep threshold fields typed in `BenchmarkPolicy`. |
| T-04-02-04 | R | benchmark CI decision surface | mitigate | Persist the policy-augmented report and map verdict failure to exit code `2` so CI sees one deterministic source of truth. |
</threat_model>
