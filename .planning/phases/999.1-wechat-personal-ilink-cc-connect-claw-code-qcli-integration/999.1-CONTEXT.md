# Phase 999.1: WeChat personal ilink -> cc-connect -> claw-code -> Qcli integration - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning

<domain>
## Phase Boundary

This backlog phase defines how FluxQ should be exposed through a personal WeChat messaging workflow using `ilink`, `cc-connect`, `claw-code`, and `Qcli`.

The scope is the integration boundary and operating model:
- how messages enter the system
- where the agent adapter lives
- how execution is routed to the remote host
- how workspaces and users are isolated
- how high-risk actions are confirmed
- what user experience the WeChat agent should present

This phase does **not** change FluxQ's core product positioning, does **not** alter the current v1.1 remote-execution roadmap sequence, and does **not** deep-embed `Qcli` into the `claw-code` runtime core.

</domain>

<decisions>
## Implementation Decisions

### Integration topology
- **D-01:** Use the direct integration route: `个人微信 -> ilink -> cc-connect(fork) -> claw adapter -> claw-code -> Qcli`.
- **D-02:** Do not introduce a separate standalone bridge service in the first version.
- **D-03:** The `cc-connect` fork should add a dedicated `claw` agent type rather than pretending to be another built-in agent type.
- **D-04:** The `claw` agent adapter should not invoke the `claw` binary directly; it should call a stable launcher entrypoint.

### Deployment and execution boundary
- **D-05:** The WeChat ingress host and the execution host are separate deployments.
- **D-06:** The `cc-connect` side should trigger the execution side through an HTTP API, not SSH.
- **D-07:** The HTTP API is a thin execution gateway in front of the local launcher, not a business-logic layer that reimplements FluxQ semantics.
- **D-08:** `Qcli` remains an independent CLI/runtime control plane and should be invoked through `qrun ... --json/--jsonl`, not embedded into `claw-code` internals.

### Capability exposure
- **D-09:** The first version should expose the full FluxQ capability set to the WeChat agent surface, including `remote submit`.
- **D-10:** The WeChat interface should be free-form conversational, not command-only.

### Risk control and confirmation
- **D-11:** Only high-risk operations require confirmation, especially `remote submit`.
- **D-12:** High-risk confirmation should use a second explicit reply from the user, not weak confirmation or silent continuation.
- **D-13:** The confirmation summary should use the **standard** format: action, input source, workspace, backend / instance, whether it may create a remote job or spend, and a short consequence statement.

### Identity, access, and workspace isolation
- **D-14:** The first version should allow only a small whitelist of approved personal WeChat users, not all contacts and not general group access.
- **D-15:** Workspace isolation should be per WeChat user, not one shared workspace and not one workspace per thread.
- **D-16:** All whitelisted users may confirm high-risk operations; authority is not restricted to one administrator in the first version.

### WeChat UX and response style
- **D-17:** The WeChat assistant should behave as a free-form agent, not a strict command shell.
- **D-18:** The default response style should be explanatory rather than terse, including background, reasoning, and risk explanation where relevant.

### the agent's Discretion
- Exact HTTP API route naming and payload envelope
- Exact launcher script path and naming convention
- Exact whitelist storage format
- Session token or conversation identifier persistence mechanism
- Exact wording and formatting of non-risk explanatory replies

</decisions>

<specifics>
## Specific Ideas

- The preferred operational model is: `cc-connect` fork owns the WeChat-side adapter, but the real execution happens remotely through a thin HTTP gateway and a local launcher on the execution host.
- The launcher should be the stable seam so later `claw` CLI flag changes do not require revisiting the `cc-connect` adapter logic.
- Although the first version is conversational, high-risk operations must still surface a strong structured confirmation summary before execution.
- The user explicitly wants the personal WeChat `ilink` route, not a fallback to WeCom / enterprise WeChat.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `docs/agent-ci-adoption.md` already describes the recommended host loop for agent and CI consumers, including `plan`, `exec`, `compare`, `doctor`, and `pack` as command boundaries.
- `docs/aionrs-integration.md` already establishes the preferred external-host pattern: files plus shell commands, not a custom internal protocol.
- `.agents/skills/fluxq-cli/SKILL.md` already encodes the expectation that host agents should stay inside the FluxQ lifecycle instead of bypassing it with ad hoc scripting.

### Established Patterns
- FluxQ integrations should consume `qrun ... --json/--jsonl` machine output instead of parsing prose.
- Host adapters should write or point to explicit intent files and then call FluxQ commands through shell invocations.
- The repository strongly favors keeping runtime-control-plane logic in `Qcli` and keeping host-specific behavior outside the runtime core.

### Integration Points
- A future `integrations/` subtree is the most natural home for launcher scripts, adapter docs, and example configs for `cc-connect`.
- `Qcli` already exposes the required command surface for the planned integration (`doctor`, `backend list`, `inspect`, `compare`, `remote submit`) and therefore does not need a platform-specific rewrite before integration planning starts.

</code_context>

<deferred>
## Deferred Ideas

- Enterprise WeCom / WeChat Work integration — intentionally out of scope because this backlog item is specifically for personal WeChat via `ilink`.
- A standalone bridge service between `cc-connect` and `claw-code` — explicitly deferred because the chosen direction is direct integration with a thin HTTP gateway only.
- Restricting high-risk confirmation authority to a single admin role — deferred because the first version allows all whitelisted users to confirm.
- Replacing free-form chat with a command-only interaction model — deferred because the chosen direction is conversational UX.

</deferred>
