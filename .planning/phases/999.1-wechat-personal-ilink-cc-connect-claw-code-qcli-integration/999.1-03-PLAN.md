<threat_model>
High-severity threats in this plan are blocking. Do not close the phase if high-risk actions can execute without explicit second-step confirmation.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| conversational turn -> high-risk tool request | Natural-language user requests can cross into actions that create remote jobs or spend |
| pending confirmation -> confirmed execution | Stored approval state is turned into a real `qrun` operation |
| operator documentation -> live deployment | Misstated setup or confirmation behavior can create unsafe operator actions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-03-01 | E/T | pending confirmation flow | mitigate | Store the exact approved tool request, require a second explicit `CONFIRM <id>` or `确认 <id>`, and block execution until that confirmation arrives. |
| T-999.1-03-02 | S/R | confirmation identity binding | mitigate | Bind each confirmation to `wechat_user.id` and `conversation.id`, and reject foreign-user or wrong-conversation confirms. |
| T-999.1-03-03 | T/D | stale or replayed confirmations | mitigate | Add TTL expiry plus one-time-use deletion on confirm or cancel; test replay and expiry failures. |
| T-999.1-03-04 | I | confirmation summary | mitigate | Generate the standard summary from the stored tool request and require tests for `action`, `input_source`, `workspace`, `backend / instance`, `may_create_remote_job`, `may_spend`, and `consequence`. |
</threat_model>
