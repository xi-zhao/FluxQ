### Locked Decisions

- The route is fixed to `个人微信 -> ilink -> cc-connect(fork) -> claw adapter -> claw-code -> Qcli`. No enterprise WeCom fallback for this phase. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- `cc-connect` should be forked and extended with a dedicated `claw` agent type instead of pretending to be an existing built-in agent integration. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- The adapter should call a stable launcher seam, not the `claw` binary directly. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- WeChat ingress and execution host are split deployments, and the ingress host should call the execution host through an HTTP API rather than SSH. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- `Qcli` remains an external runtime control plane invoked through `qrun ... --json/--jsonl`, not embedded into the `claw-code` core. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`; `docs/agent-ci-adoption.md`; `docs/aionrs-integration.md`]
- The first version exposes the full FluxQ capability set, including `remote submit`, but high-risk operations must use explicit second-step confirmation with a standard confirmation summary. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- Access is limited to a small allowlist; workspace isolation is per WeChat user; all allowlisted users may confirm high-risk operations in the first version. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- The WeChat UX is conversational and explanatory, not command-only and not terse-by-default. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]

## Summary

This phase is not about extending FluxQ runtime semantics. It is about wrapping the existing FluxQ control plane in a trustworthy chat ingress for personal WeChat, with `cc-connect` as the messaging gateway and `claw-code` as the agent host. The repository already has the right product posture for that: both `docs/agent-ci-adoption.md` and `docs/aionrs-integration.md` explicitly say host systems should drive FluxQ through shell commands and machine-readable JSON/JSONL instead of embedding runtime logic into the host. [VERIFIED: `docs/agent-ci-adoption.md`; `docs/aionrs-integration.md`]

The external platform facts line up with the chosen route:

- `cc-connect` currently supports personal Weixin through `ilink` HTTP long polling and documents a one-command setup flow that writes `token`, `base_url`, and `account_id` back into `config.toml`. [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/README.md] [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/docs/weixin.md]
- `cc-connect` ships built-in agent types for several CLIs, but its config example shows provider routing by `agent_types` and does not list a native `claw` adapter. That means Phase 999.1 really does need a fork-side `claw` agent implementation instead of pure configuration. [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/config.example.toml]
- `claw-code` is build-from-source-first, does not yet ship a real ACP daemon entrypoint, and its documented reliable scripting surface today is the CLI binary itself (`claw prompt`, `claw doctor`, `--output-format json`, `--resume latest`, permission flags). That reinforces the decision to integrate through a launcher and thin HTTP gateway instead of expecting daemon-native attachment. [CITED: https://raw.githubusercontent.com/ultraworkers/claw-code/main/README.md] [CITED: https://raw.githubusercontent.com/ultraworkers/claw-code/main/USAGE.md]

**Primary recommendation:** plan the work in three layers:

1. `cc-connect` fork changes that add a `claw` agent type plus message-side allowlist and confirmation hooks
2. a thin remote execution gateway + launcher seam that normalizes WeChat messages into stable `claw` invocations
3. FluxQ command wrappers / prompt guidance that keep `Qcli` external and machine-readable

That split matches the locked boundary and avoids mixing transport concerns, session concerns, and FluxQ runtime concerns in one module. [ASSUMED]

[CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/README.md] [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/config.example.toml] |

| `Qcli` / `qrun` | Existing repo docs and CLI | Runtime control plane, machine-readable commands, remote execution control | This is already the product core and should remain the execution authority. [VERIFIED: `.planning/PROJECT.md`; `docs/agent-ci-adoption.md`; `docs/aionrs-integration.md`] |

### Supporting

| Component | Purpose | When to Use |
|-----------|---------|-------------|
| Launcher script | Stable local seam between gateway and `claw` CLI | Always. Prevents `cc-connect` adapter code from hardcoding `claw` CLI details. |
| Thin HTTP gateway | Remote trigger boundary between ingress host and execution host | Always in the chosen split-deploy topology. It should own auth, request normalization, and response streaming/collection only. |
| FluxQ wrappers | Stable subcommands or scripts that call `qrun ... --json/--jsonl` | Use for high-signal capabilities (`doctor`, `backend list`, `inspect`, `compare`, `remote submit`) so `claw` does not need to invent shell syntax each time. |
| Allowlist + per-user workspace resolver | Maps WeChat identity to authorization and workspace root | Required because workspace isolation is per WeChat user and the route is conversational rather than command-only. |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Forking `cc-connect` to add `claw` agent type | Pretend `claw` is `codex` / `claudecode` compatible | Brittle and likely to break on output/protocol drift. The docs do not show `claw` as a built-in type today. [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/config.example.toml] |
| Thin HTTP gateway | SSH into execution host and run launcher remotely | Simpler to prototype, but poor fit for conversational session continuity, structured error envelopes, and long-lived per-user state. [ASSUMED] |
| External bridge service | Direct fork-side adapter with launcher seam | An external bridge is cleaner in the abstract, but the locked decision is direct integration, and a separate service would add deployment and observability overhead too early. ## Architecture Patterns

### Recommended Project Structure

```text
integrations/
├── cc-connect/
│   ├── README.md                  # operator-facing integration guide
│   ├── config.toml.example        # example project/platform config for cc-connect
│   ├── prompts/
│   │   └── claw-system-prompt.md  # WeChat-facing behavioral policy for claw
│   ├── bin/
│   │   ├── run-claw-agent         # stable launcher seam called by the gateway
│   │   ├── fluxq-doctor
│   │   ├── fluxq-backend-list
│   │   ├── fluxq-inspect
│   │   ├── fluxq-compare
│   │   └── fluxq-remote-submit
│   └── gateway/
│       ├── README.md              # HTTP contract and deployment notes
│       └── example.env            # gateway auth / routing variables
docs/
├── agent-ci-adoption.md
└── aionrs-integration.md
```

This repo already uses `integrations/` as the natural place for host-specific guidance, so the first implementation slice should live there too. [VERIFIED: `integrations/aionrs/CLAUDE.md.example`; `integrations/aionrs/hooks.example.toml`]

### Pattern 1: Messaging Platform Owns Transport, FluxQ Owns Runtime

`cc-connect` should own:
- Weixin polling / send operations
- project/platform routing
- chat identity and allowlist decisions
- conversation/session bookkeeping on the ingress side

FluxQ should own:
- runtime state
- compare / doctor / inspect / remote submit semantics
- machine-readable result contracts

The gateway/launcher seam exists precisely so these boundaries stay clear. [VERIFIED: `docs/agent-ci-adoption.md`; `docs/aionrs-integration.md`] [ASSUMED]

### Pattern 2: Conversation -> Gateway -> Launcher -> CLI, Not Conversation -> Runtime Directly

The direct route should still be layered:

1. WeChat message arrives in `cc-connect`
2. `cc-connect` adapter performs allowlist check and conversation classification
3. Adapter calls remote HTTP gateway
4. Gateway resolves user -> workspace and confirmation policy
5. Gateway invokes launcher
6. Launcher invokes `claw`
7. `claw` invokes FluxQ wrappers / `qrun`

This keeps the `cc-connect` fork thin and gives the split deployment a stable RPC seam. [ASSUMED]

### Pattern 3: Conversational Frontend, Structured Risk Gate

Because the chosen UX is conversational and explanatory, intent detection can stay natural-language-first. But high-risk actions must convert into a structured confirmation state before execution.

Minimum confirmation payload should include:
- action
- source kind / user intent summary
- workspace
- backend
- instance
- spend / remote-job warning
- consequence statement

The user then confirms in a second explicit reply. This is the narrowest trustworthy path compatible with the chosen UX. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]

### Pattern 4: Per-User Workspace Resolution

The integration must resolve every WeChat identity to a deterministic FluxQ workspace root. A good first pattern is:

```text
~/.cc-connect/projects/fluxq/workspaces/<wechat-user-id>/.quantum
```

or an equivalent per-user directory on the execution host. The important property is that one user cannot silently reuse another user's runtime history or baseline state. [ASSUMED]

### Pattern 5: Launcher As the Stable seam

The launcher should own:
- environment setup for `claw`
- model / permission defaults
- system prompt injection
- wrapper path exposure
- log path / stdout-stderr normalization

`cc-connect` should only know "call launcher with request payload", not the internal `claw` CLI flags. That is the main leverage of D-02. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`] [ASSUMED]

## External Constraints and Risks

### `cc-connect` / personal Weixin

- Personal Weixin support is explicitly documented as a Weixin/ilink long-polling route. That is good for "no public IP" ingress, but it also means this phase inherits whatever reliability and auth semantics `ilink` exposes. [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/README.md] [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/docs/weixin.md]
- `cc-connect` supports privileged users per project via `admin_from`, but that is not yet equivalent to the per-user workspace + per-risk confirmation model chosen here. Planning should assume custom glue is needed. [CITED: https://raw.githubusercontent.com/chenhg5/cc-connect/main/README.md]

### `claw-code`

- `claw-code` does not ship a real ACP daemon yet, so "daemon-native adapter" should not be the Phase 999.1 assumption. [CITED: https://raw.githubusercontent.com/ultraworkers/claw-code/main/README.md]
- `claw` does expose JSON output for diagnostic verbs and prompt one-shots, plus session resume and permission flags. The plan should treat that as the stable starting point. [CITED: https://raw.githubusercontent.com/ultraworkers/claw-code/main/USAGE.md]

### FluxQ product boundary

- FluxQ is not the chat platform. This integration should remain clearly subordinate to the runtime control plane and reuse machine-readable `qrun` surfaces instead of building WeChat-specific logic into the core CLI. [VERIFIED: `.planning/PROJECT.md`; `docs/agent-ci-adoption.md`] 

## Validation Architecture

This phase needs verification at three levels:

1. **Static/contract tests**
   - configuration parsing
   - allowlist and workspace-resolution logic
   - confirmation-state machine behavior
   - launcher command construction

2. **Gateway integration tests**
   - `cc-connect` adapter -> HTTP gateway request
   - HTTP gateway -> launcher invocation
   - launcher -> `claw` invocation
   - mocked `qrun` JSON/JSONL passthrough

3. **Operator smoke tests**
   - personal Weixin message enters the system
   - agent replies over the same route
   - confirmation flow for `remote submit` works end-to-end

The planning phase should include a fake-provider / fake-launcher lane first, because this integration crosses too many external seams to rely on only live tests.

## Recommended Planning Split

The first executable plan set should likely split into **three plans**:

1. **Transport and adapter contract**
   - define `cc-connect` fork assumptions
   - define HTTP gateway request/response contract
   - define per-user workspace resolution and allowlist handling

2. **Launcher and FluxQ tool surface**
   - create launcher seam
   - create FluxQ wrapper commands
   - define `claw` prompt / tool policy for WeChat

3. **Risk controls and operator UX**
   - high-risk confirmation state machine
   - explanatory reply formatting
   - end-to-end mocked tests and operator docs

That split matches the locked decisions and keeps transport, execution, and UX/risk concerns independently testable. [ASSUMED]

## Anti-Patterns to Avoid

- **Do not embed FluxQ runtime semantics into the `cc-connect` fork.** That would violate the existing host-integration guidance and make the messaging fork a second runtime surface. [VERIFIED: `docs/agent-ci-adoption.md`; `docs/aionrs-integration.md`]
- **Do not assume ACP support in `claw-code`.** The upstream docs explicitly say real ACP daemon support is not there yet. [CITED: https://raw.githubusercontent.com/ultraworkers/claw-code/main/README.md]
- **Do not let conversational UX silently bypass confirmation for `remote submit`.** That would be the highest-risk conflict with the chosen free-form interface. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
- **Do not use a shared workspace for all WeChat users.** The locked decision is per-user isolation, and FluxQ history/baselines are too stateful for a global chat-driven workspace. [VERIFIED: `.planning/phases/999.1-wechat-personal-ilink-cc-connect-claw-code-qcli-integration/999.1-CONTEXT.md`]
