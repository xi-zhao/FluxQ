<threat_model>
High-severity threats in this plan are blocking. Do not mark the plan complete unless the mitigations below are implemented and covered by tests.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| personal WeChat ingress -> execution gateway | Untrusted network traffic crosses from the `cc-connect` fork into the execution host |
| allowlisted WeChat identity -> workspace root | A chat identity is translated into a filesystem execution root that must remain isolated per user |
| signed HTTP request -> launcher request envelope | Gateway-authenticated message payloads become launcher work items |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-01-01 | S/E | `POST /v1/integrations/wechat/turn` | mitigate | Require HMAC headers, timestamp skew validation, and duplicate-nonce rejection before any launcher call. |
| T-999.1-01-02 | T/E | workspace mapping | mitigate | Ignore caller-supplied workspace paths; derive `<workspaces_root>/<workspace_key>/.quantum` only from the server-side allowlist mapping. |
| T-999.1-01-03 | I | gateway response envelope | mitigate | Keep the gateway thin: pass `reason_codes`, `next_actions`, and `gate` through, and do not serialize raw secrets or ad hoc shell data. |
| T-999.1-01-04 | D | replay protection | mitigate | Persist a recent nonce cache under `FLUXQ_GATEWAY_STATE_ROOT` and block stale or duplicate signed requests. |
</threat_model>
