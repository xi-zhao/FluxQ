<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| canonical submit input -> remote attempt store | Canonical `QSpec` and input provenance cross into a new durable workspace namespace |
| workspace manifest -> remote attempt counter | New attempt identity is allocated inside the same workspace-lock and atomic-write regime as revision identity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-01 | T | attempt identity vs revision identity | mitigate | Keep `current_attempt` additive and separate from `current_revision`; test that submit persistence never bumps the revision counter. |
| T-10-01-02 | I | remote attempt record | mitigate | Model only non-secret metadata (`auth_source`, `backend`, `instance`, `job_id`) and reject token-bearing fields in typed models and tests. |
| T-10-01-03 | D | latest alias promotion | mitigate | Use atomic writes for history + latest alias and test that interrupted writes cannot leave mixed authoritative pointers. |
| T-10-01-04 | E | workspace persistence surface | mitigate | Reuse `acquire_workspace_lock()` and the workspace handle instead of writing remote attempt files outside the lock discipline. |
</threat_model>
