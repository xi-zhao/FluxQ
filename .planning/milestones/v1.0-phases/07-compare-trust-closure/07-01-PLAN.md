<objective>
修复 producer 侧 canonical revision write seam，让第二次 `exec` 写出的 report/qspec/manifest 重新保持自洽。

Purpose: Phase 07 的阻塞点不是 compare policy 语义缺失，而是 writer 在 alias promotion 之前把 mutable alias 回灌进 canonical history，导致 trusted import 在 compare 之前就 fail closed。本计划必须修 producer，不得通过削弱 compare/import 的 fail-closed 行为来“放行”错误 revision。范围只覆盖修复后的新写入路径与 fresh/healthy workspace；**已经被旧 bug 污染的历史 workspace 不在本计划自动修复范围内**。
Output: history-only 的 report 写入契约、保持 history-first 的 exec 提交流程、以及能在第二次 `exec` 后抓住 revision coherence 的红绿回归。
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| exec producer -> canonical history artifacts | `exec` 产出的 qspec/report/manifest 会被后续 compare/import 视为可信 revision truth layer。 |
| report payload -> replay integrity consumer | `resolve_workspace_current()` 依赖 report 内的 qspec hash/semantic metadata 来决定是否 fail closed。 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01-01 | T | `src/quantum_runtime/reporters/writer.py` | mitigate | 禁止 alias -> history 回写；writer 只消费 executor 已写好的 history 路径，并用回归测试锁住第二次 exec 的 qspec/report 一致性。 |
| T-07-01-02 | R | report `qspec.hash` / `qspec.semantic_hash` contract | mitigate | 从同一个 canonical qspec history 文件同时导出 hash 与 semantic hash，避免 report 记录不可追责地混用两个来源。 |
</threat_model>
