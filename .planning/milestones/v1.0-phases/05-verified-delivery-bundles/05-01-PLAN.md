<objective>
Harden `qrun pack` into a path-safe, non-destructive delivery producer for DELV-01.

Purpose: the current pack flow proves FluxQ can copy a revision into a pack directory, but it is still too destructive and too workspace-dependent for a delivery boundary. This plan turns pack into a read-only snapshot over immutable revision artifacts, adds bundle-local trust metadata, and closes the `--revision` path-safety gap before verification or import build on top of it.
Output: a bundle-local manifest, strict revision validation, revision-scoped `events.jsonl` plus `trace.ndjson`, and a pack flow that promotes staged bundles only after validation passes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI `--revision` input -> pack filesystem paths | Raw revision text controls which bundle directory will be staged, replaced, or rejected. |
| immutable revision history -> portable bundle bytes | Durable runtime evidence crosses into a copied bundle that downstream tools will treat as authoritative. |
| staged bundle -> promoted `packs/<revision>` directory | A failed staged verification must not destroy the last known-good bundle. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01-01 | T | `qrun pack --revision` path derivation | mitigate | Reuse one shared `invalid_revision` validator before deriving `packs/<revision>` and assert all derived paths stay under `WorkspacePaths.packs_dir`. |
| T-05-01-02 | T/R | bundle-local trust metadata | mitigate | Add `bundle_manifest.json` with relative paths and SHA-256 digests computed from the staged bundle, not from mutable workspace aliases. |
| T-05-01-03 | D | repack promotion flow | mitigate | Keep the previous bundle until staged verification passes, then replace atomically. |
</threat_model>
