# FluxQ v0.2.3

FluxQ `v0.2.3` turns this release into a clearer decision-grade local runtime release: baseline compare, trust-honest benchmark reporting, bounded parameterized expectation evaluation, and explicit dependency diagnostics for coding agents and CI loops.

## Why This Release Matters

The product story is tighter than in `0.2.0`. FluxQ is no longer just a reproducible local quantum workflow CLI. It now has a more explicit decision path: establish a baseline, compare the current run against that baseline, inspect export and benchmark provenance, and keep dependency drift honest before continuing.

## Highlights

- workspace baseline persistence with `qrun baseline set/show/clear`
- `qrun compare --baseline --json` for approval-style agent and CI checks
- explicit export provenance and replay-integrity reporting across compare and inspect
- target-aware benchmark honesty with comparability and fallback metadata
- optional backends stay advisory until a workspace actually depends on them
- parameter workflows are bounded local expectation evaluation, not a general optimizer

## Install

For direct CLI use from GitHub:

```bash
uv tool install git+https://github.com/xi-zhao/FluxQ@v0.2.3
```

For local development:

```bash
uv venv --python 3.11
source .venv/bin/activate
uv pip install -e '.[dev,qiskit]'
```

## What To Try First

```bash
qrun init --workspace .quantum --json
qrun exec --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun baseline set --workspace .quantum --revision rev_000001 --json
qrun compare --workspace .quantum --baseline --json
qrun bench --workspace .quantum --json
qrun doctor --workspace .quantum --json --fix
```

## Best Fit

FluxQ is a strong fit if you want:

- reproducible quantum workflow state instead of disposable generated scripts
- approval-style baseline comparison for agent-generated revisions
- benchmark outputs that explain when they are comparable and when they are not
- parameter workflow support that stays explicit about being local expectation evaluation rather than a full optimizer stack
