# FluxQ v0.3.0

FluxQ `v0.3.0` is the runtime control-plane release: the point where FluxQ becomes an explicit agent-first quantum runtime CLI with a runtime control plane rather than just a local workflow CLI with natural-language ingress.

## Why This Release Matters

Earlier releases established the trusted local runtime loop: `intent -> qspec -> report -> compare -> continue`.

`0.3.0` makes that loop easier for agents and CI systems to consume directly by adding immutable run manifests, schema-versioned JSON payloads, and read-mostly control-plane commands that answer "what would run?", "what is the workspace state?", and "what exactly happened in this revision?" without scraping human-oriented stdout.

## Highlights

- new `qrun plan` dry-run command for resolving one input into a machine-readable execution plan
- new `qrun status` workspace summary for readiness, active artifacts, and baseline state
- new `qrun show` run view for latest or historical revisions
- new `qrun schema` output for public runtime contracts
- immutable `manifests/history/<revision>.json` plus `manifests/latest.json`
- `schema_version` on machine-readable command payloads and core runtime artifacts
- structured `error_code` and `remediation` fields for JSON-mode CLI failures

## Install

For direct CLI use from GitHub:

```bash
uv tool install git+https://github.com/xi-zhao/FluxQ@v0.3.0
```

This install includes the local `qiskit-local` runtime dependencies by default. `classiq` remains optional.

For local development:

```bash
uv venv --python 3.11
source .venv/bin/activate
uv pip install -e '.[dev]'
```

## What To Try First

```bash
qrun init --workspace .quantum --json
qrun plan --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun exec --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun baseline set --workspace .quantum --revision rev_000001 --json
qrun status --workspace .quantum --json
qrun show --workspace .quantum --json
qrun compare --workspace .quantum --baseline --json
qrun schema manifest
```

## Best Fit

FluxQ is a strong fit if you want:

- an agent-first quantum runtime CLI rather than a chat-shaped quantum tool
- canonical `QSpec` state instead of prompt text as the truth layer
- immutable run objects with provenance, revision history, and compare-ready semantics
- file-based JSON contracts and stable exit codes that CI and coding agents can orchestrate directly
