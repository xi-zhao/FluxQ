# FluxQ v0.2.0

FluxQ `v0.2.0` is the first public release baseline for a workspace-native quantum workflow runtime aimed at coding agents, CI, and revision-aware quantum development loops.

## Why This Release Matters

Most quantum tooling still treats generation as a one-shot export problem. FluxQ takes a different stance: a useful runtime should preserve intent, QSpec, artifacts, provenance, and replay trust across revisions so that teams and agents can compare, inspect, and continue work instead of starting from scratch every time.

## Highlights

- deterministic `.quantum` workspaces with revision-stable history
- replayable reports with explicit provenance and replay-integrity checks
- semantic workload comparison across revisions and copied reports
- Qiskit, OpenQASM 3, and Classiq Python lowering paths
- local simulation, transpile validation, diagrams, and structural benchmarking
- agent/CI-friendly JSON outputs for `exec`, `inspect`, `export`, `bench`, `compare`, and `doctor`

## Install

For direct CLI use from GitHub:

```bash
uv tool install git+https://github.com/xi-zhao/FluxQ@v0.2.0
```

For local development:

```bash
uv venv --python 3.11
source .venv/bin/activate
uv pip install -e '.[dev,qiskit]'
```

## What To Try First

```bash
qrun init --workspace .quantum --json
qrun exec --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun inspect --workspace .quantum --json
qrun compare --workspace .quantum --left-revision rev_000001 --right-revision rev_000002 --expect same-subject --json
```

## Best Fit

FluxQ is a strong fit if you want:

- reproducible quantum workflow state instead of disposable generated scripts
- file-based orchestration for coding agents and CI
- export, diagnostics, and trust signals tied to the same revisioned workspace
- a base runtime that can grow toward stronger benchmark and backend workflows over time
