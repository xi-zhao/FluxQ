# FluxQ v0.3.1

FluxQ `v0.3.1` is the agent-observability release for the runtime control plane.

## Why This Release Matters

`0.3.0` made FluxQ a machine-first runtime control plane. `0.3.1` makes that surface easier for agents and CI loops to consume incrementally by adding shared decision signals, normalized ingress, persisted runtime objects, and JSONL event streams.

Agents no longer need to open multiple workspace files just to infer whether a run is healthy, what failed, or what command should happen next.

## Highlights

- `prompt` and `resolve` as the ingress boundary, with `qspec` as the truth layer
- persisted `intent.json`, `plan.json`, `qspec.json`, `report.json`, and `manifest.json` revision artifacts
- `health`, `reason_codes`, `next_actions`, and `decision` or `gate` blocks on the control-plane JSON surface
- `--jsonl` event streams for `qrun exec`, `qrun compare`, `qrun bench`, and `qrun doctor`
- canonical `events.jsonl` workspace events alongside the append-only trace log
- export profiles such as `qiskit-native` and `qasm3-generic`
- `qrun pack` for a portable revision bundle
- terminal JSONL events that carry the same core machine payload as the one-shot `--json` commands
- stricter active artifact trust checks for `qspec`, `report`, `manifest`, and baseline state
- safer copied-report relocation when a caller-provided workspace can satisfy the expected revision inputs

## Install

For direct CLI use from GitHub:

```bash
uv tool install git+https://github.com/xi-zhao/FluxQ@v0.3.1
```

This install includes the local `qiskit-local` runtime dependencies by default. `classiq` remains optional.

For local development:

```bash
uv venv --python 3.11
source .venv/bin/activate
uv pip install -e '.[dev]'
```

## What To Try First

```bash
qrun prompt "Build a 4-qubit GHZ circuit and measure all qubits." --json
qrun resolve --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun init --workspace .quantum --json
qrun plan --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun exec --workspace .quantum --intent-file examples/intent-ghz.md --jsonl
qrun baseline set --workspace .quantum --revision rev_000001 --json
qrun compare --workspace .quantum --baseline --fail-on subject_drift --json
qrun doctor --workspace .quantum --json --ci
qrun pack --workspace .quantum --revision rev_000001 --json
qrun pack-inspect --pack-root .quantum/packs/rev_000001 --json
qrun pack-import --pack-root .quantum/packs/rev_000001 --workspace downstream/.quantum --json
```

## Runtime Contract Stability

- Stable runtime contracts: `QSpec.version` stays explicit in serialized specs, CLI/result/artifact `schema_version` stays separate from `QSpec.version`, workspace history remains append-only under `.quantum/*/history/`, and verified delivery handoff runs through `qrun pack`, `qrun pack-inspect`, and `qrun pack-import`.
- Evolving runtime contracts: `reason_codes`, `next_actions`, `decision`, and `gate` remain additive machine signals, and released CLI JSON responses can add fields without renaming or removing released fields.
- Optional runtime contracts: `classiq`, `docs/aionrs-integration.md`, `integrations/aionrs/hooks.example.toml`, and export profiles beyond the base local path remain capability-gated adoption surfaces.
- Safe consumption rule: automate against the stable contracts directly, consume evolving JSON by presence-checking additive fields, and treat optional integrations as opt-in surfaces gated by installed capabilities and release-line guidance.

## Best Fit

FluxQ is a strong fit if you want:

- a runtime control plane with additive machine signals instead of human-only stdout
- event streams for long-running quantum workflow steps in agent loops
- explicit next-step hints for CI and coding agents
- portable report-backed workflows that remain conservative about trust and integrity
