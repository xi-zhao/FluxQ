# FluxQ v0.2.4

FluxQ `v0.2.4` is a patch release that tightens the public install and trust contract around the decision-grade local runtime surface introduced in `0.2.3`.

## Why This Release Matters

`0.2.3` established the right workflow shape, but its public install and release story still had a mismatch: the documented GitHub install path did not match the actual runtime dependency surface, default benchmarking could over-report an optional backend the active QSpec never requested, and current-workspace export could silently drop provenance after an integrity mismatch.

`0.2.4` fixes those release-surface gaps so the documented first-run path is honest again.

## Highlights

- public `qrun` install now includes the local `qiskit-local` runtime dependencies by default
- default `qrun bench --json` runs `qiskit-local` and only adds optional backends the active QSpec explicitly requests
- `qrun export --json` now fails closed on current-workspace replay/provenance mismatches instead of silently nulling `source_*` fields
- optional backends stay advisory until a workspace actually depends on them
- parameter workflows are bounded local expectation evaluation, not a general optimizer

## Install

For direct CLI use from GitHub:

```bash
uv tool install git+https://github.com/xi-zhao/FluxQ@v0.2.4
```

This install includes the local `qiskit-local` runtime dependencies by default. `classiq` remains optional.

For local development:

```bash
uv venv --python 3.11
source .venv/bin/activate
uv pip install -e '.[dev]'
```

## What To Try First

```bash
qrun init --workspace .quantum --json
qrun exec --workspace .quantum --intent-file examples/intent-ghz.md --json
qrun baseline set --workspace .quantum --revision rev_000001 --json
qrun compare --workspace .quantum --baseline --json
qrun bench --workspace .quantum --json
qrun doctor --workspace .quantum --json --fix
```

By default, `qrun bench` runs `qiskit-local` and only adds optional backends that the active QSpec explicitly requests.

## Best Fit

FluxQ is a strong fit if you want:

- reproducible quantum workflow state instead of disposable generated scripts
- approval-style baseline comparison for agent-generated revisions
- benchmark outputs that explain when they are comparable and when they are not
- parameter workflow support that stays explicit about being local expectation evaluation rather than a full optimizer stack
