# Post-v0.2 Feature Roadmap

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Turn FluxQ from a released quantum workflow CLI into a decision-grade runtime that serious quantum developers, coding agents, and CI systems can rely on for baseline comparison, target-aware benchmarking, and parameterized workflow iteration.

**Architecture:** The next phase should deepen the released runtime loop rather than widen the backend matrix. The product wedge is a trusted local workflow substrate: `intent -> qspec -> artifacts -> diagnostics -> report -> compare -> continue`. The next build batch should strengthen that loop in three directions: baseline workflows, target-aware backend comparison, and parameterized expectation-value workflows.

**Tech Stack:** Python 3.11, Typer CLI, Pydantic QSpec/report models, Qiskit, Classiq integration, pytest, GitHub release workflow

## Why This Is The Right Next Phase

### CEO view

- FluxQ should become the decision layer for CI and agents, not just the generator that runs before the decision.
- The strongest near-term differentiation is trusted baseline comparison and provenance-backed backend comparison.
- Remote submit and provider sprawl would dilute the product before the local runtime loop is dominant.

### Quantum expert view

- Serious quantum users need parameter binding, sweeps, and expectation-value workflows rather than only static circuit snapshots.
- Benchmark credibility improves most when compilation and synthesis metrics are target-aware and explicitly comparable.
- Long-tail pattern expansion is lower value than strengthening problem semantics, observables, and target-aware diagnostics.

## Strategic Priority Order

1. Baseline workflows and policy-driven compare
2. Target-aware backend comparison and benchmark parity
3. Parameterized expectation-value workflows
4. Broader problem-native workflow semantics
5. Remote submit design only after the above are stable

## Non-Goals For This Phase

- do not add more backend vendors just to widen the matrix
- do not ship remote execution or job orchestration yet
- do not expand into a large pattern catalog before deepening semantics on the existing high-value workflows
- do not present Qiskit transpilation and Classiq synthesis metrics as directly equivalent without explicit benchmark mode and target assumptions

## Six-Week Sequence

### Week 1: Baseline contract

Define the product contract for baseline-driven workflows.

Deliverables:

- stable concept of `baseline` as a first-class comparison input
- clear policy model for semantic drift, replay-integrity regressions, and backend regressions
- schema notes for how reports and compare results reference baselines
- end-to-end examples that make the CI story obvious

Exit criteria:

- a developer can explain, without reading code, how FluxQ decides whether a new run is acceptable versus the approved baseline

### Week 2: Baseline implementation

Ship baseline-aware compare flows and JSON verdicts.

Deliverables:

- CLI support for baseline references
- machine-readable verdicts with failed checks and highlights
- workspace-friendly baseline persistence or references
- tests covering happy path, semantic drift, replay-trust regression, and detached inputs

Exit criteria:

- FluxQ can gate a CI run against an approved baseline using released JSON fields only

### Week 3: Baselines across reports and inspect

Extend baseline-awareness into the wider runtime surface.

Deliverables:

- report JSON that explains baseline lineage where relevant
- inspect output that exposes baseline/trust status cleanly
- docs and examples for baseline-first agent and CI usage

Exit criteria:

- baseline semantics feel like part of the runtime, not a compare-only bolt-on

### Week 4: Target-aware backend comparison

Strengthen trusted backend comparison with real target assumptions.

Deliverables:

- target-aware Qiskit transpilation inputs using target/coupling assumptions
- clearer Classiq synthesis extraction and fallback markers
- benchmark outputs that distinguish structural, transpiled, and synthesis-backed modes
- normalized backend capability metadata for target-aware comparison

Exit criteria:

- a user can tell what target assumptions drove a benchmark and whether the reported metrics are truly comparable

### Week 5: Parameterized expectation-value workflows

Move the runtime beyond static circuit snapshots.

Deliverables:

- parameter bind or sweep inputs for `hardware_efficient_ansatz` and `qaoa_ansatz`
- observable or expectation-value reporting for supported workflows
- reports and diagnostics that expose parameter and observable semantics cleanly
- examples that feel relevant to actual iterative quantum prototyping

Exit criteria:

- FluxQ can evaluate a parameterized workflow in a way that is meaningfully closer to real algorithm iteration than to a demo circuit export

### Week 6: Problem-native workflow depth and release prep

Use the final week to harden and package the phase.

Deliverables:

- strengthen generalized graph QAOA or Pauli-evolution style workflow semantics where they fit the released IR
- finalize docs, examples, release notes, and CI guidance around three flagship stories:
  - baseline guardrails
  - target-aware backend comparison
  - parameterized expectation-value workflows
- prepare the next `0.2.x` release from this completed surface

Exit criteria:

- the next release tells a coherent product story and expands real utility, not just command count

## Release Recommendation

Do not jump to `0.3.x` yet.

Recommended release path:

- `0.2.1`: baseline workflows and policy-driven compare
- `0.2.2`: target-aware benchmark parity
- `0.2.3`: parameterized expectation-value workflows

Only after those three are stable should FluxQ start the design-heavy remote-submit phase.

## Decision Rule

If the next feature does not improve at least one of these, it is probably not the right next move:

1. baseline-driven decision making
2. target-aware technical credibility
3. iterative quantum workflow usefulness
